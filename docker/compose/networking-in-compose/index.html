默认情况下 Compose 为你的 app 设置单独一个 network

服务的每个容器都加入默认网络, 并且可以被网络上的其他容器访问, 并且可以通过与容器名称相同的主机名被它们发现

注意: 你的 app 的网络被赋予基于项目名称的名称, 这个网络名称基于 app 所在的目录名称. 可以通过 --project-name 标志或 COMPOSE_PROJECT_NAME 环境变量来覆盖项目名称

当运行 docker-compose up 时, 会发生如下事情:

  1. 创建了一个名称为 myapp_default 的网络
  2. 使用 web 配置创建了一个容器. 这个容器使用名称 web 加入到网络 myapp_default 中
  3. 使用 db 配置创建了一个容器. 这个容器使用名称 db 加入到网络 myapp_default 中

注意: 在 v2.1+ 中 overlay 网络总是可以 attachable 的, 且不可配置, 这意味着独立容器可以连接到 overlay 网络; 在 3.x Compose 文件中, 可以可选地设置 attachable 属性为 false

每个容器现在可以查找主机名 web 和 db 并取回适当的容器 IP 地址

例如, web 应用代码可以连接到 postgres://db:5432 URL 并开始使用 Postgres 数据库

分清楚 HOST_PORT 和 CONTAINER_PORT 是很重要的

在上面的例子中, 对于 db, HOST_PORT 是 8081 且容器端口是 5432 (postgres 默认端口)

网络中的服务到服务通信使用 CONTAINER_PORT

如果定义了 HOST_PORT, 那么服务也可以在 swarm 之外访问

在 web 容器中, 连接到 db 的连接字符串看起来像这样 postgres://db:5432; 并且在宿主机上, 连接字符串看起来像这样 postgres://{DOCKER_IP}:8081

Update containers (更新容器)

如果修改了服务配置并运行 docker-compose up 来更新这个服务, 那么旧容器就会被删除, 新容器会使用不同的 IP 地址加入到网络中, 但是新容器继续使用相同的服务名称

运行中的容器可以查找该名称并连接到新地址, 但是旧地址停止工作

如果任何容器有到旧容器打开的连接, 这些连接将会被关闭. 这是容器的责任去检测连接, 重新查找名称并重新连接

Links (连接)

连接允许你定义额外的别名, 通过这些别名可以从另一个服务访问服务

它们不需要启用服务进行通信 - 默认情况下, 任何服务都可以使用该服务的名称访问任何其他服务
